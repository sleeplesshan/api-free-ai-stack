# 생성형 AI 웹 UI 자동화 및 응답 구조화 기술 정리 (API 무과금 연동)

## 문서 목적

이 문서는 프로그램 내에서 생성형 AI(ChatGPT, Claude, Gemini, Copilot 등) 기능을 활용할 때, **비용이 발생하는 공식 API를 사용하는 대신 AI 서비스의 무료 웹 UI를 프로그래밍적으로 제어(RPA 방식)하여 연동**하는 기술 스택과 아키텍처를 정리한 문서입니다.

특정 프레임워크나 AI 서비스에 종속되지 않는 범용적인 웹 자동화 패턴을 다룹니다. 핵심 목표는 아래와 같습니다.

1. **비용 절감:** 종량제 API 과금 없이 무료 웹 채팅 UI를 통해 AI 추론 결과를 얻어낸다.
2. **프로세스 자동화:** 사용자가 직접 AI 창에 복사/붙여넣기 하지 않아도 백그라운드에서 질의를 자동화한다.
3. **데이터 구조화:** AI의 답변을 사람이 읽는 문장 대신 `JSON`으로 강제하여 다른 프로그램에서 즉시 파싱/재사용한다.
4. **안정성 확보:** 웹 UI 변화나 봇 탐지(Bot Detection)로 인해 자동화가 깨질 수 있으므로, 실패 시 시스템이 멈추지 않는 수동 Fallback 경로를 반드시 둔다.

---

## 이 기술이 해결하는 문제

API 대신 웹 UI를 브라우저 제어 툴(Playwright, Selenium 등)로 조작하여 답변을 추출하려면 아래와 같은 까다로운 문제들이 생깁니다.

- AI는 기본적으로 사람이 읽는 자연어 텍스트와 마크다운 섞인 답변을 준다.
- 웹 화면의 전체 텍스트를 크롤링하면 메뉴, 사이드바, 이전 대화 목록까지 같이 딸려온다.
- 로그인 세션과 브라우저 세션이 다르면 인증 단계에서 곧바로 실패한다.
- 프롬프트를 입력해도 UI 스크립트 지연으로 전송이 안 되거나, 응답이 완료되기 전에 임시 상태(Placeholder)를 읽어버린다.
- 응답이 완벽한 JSON이 아닐 때 파싱이 실패한다.

따라서 실전 수준의 웹 자동화는 단순히 "브라우저 열기"가 아니라 아래 5단계를 모두 포함해야 합니다.

1. 인증 가능한 전용 브라우저 세션 준비
2. 프롬프트를 JSON 계약 형태로 설계
3. 동적으로 변하는 AI 입력창 탐지 및 전송 자동화
4. 스트리밍 응답이 실제 완료될 때까지 대기
5. 응답 본문에서 필요한 JSON 블록만 선별 및 파싱

## 전체 구조

이 기술은 아래 흐름으로 동작합니다. 어떤 업무(메일 요약, 공지 분류, 회의록 추출 등)든 이 파이프라인을 동일하게 태울 수 있습니다.

```
원본 업무 데이터 수집
→ AI 지시용 프롬프트 조합 (JSON 스키마 포함)
→ 브라우저 제어 툴(Playwright 등)로 대상 AI 웹 페이지 열기
→ 채팅 입력창 DOM 탐지
→ 프롬프트 입력 및 전송
→ AI 응답 스트리밍 완료 대기
→ 최신 응답 텍스트 추출
→ 텍스트 내에서 후보 JSON 객체 탐색
→ 조건에 맞는 의미 있는(meaningful) JSON만 선별
→ 프로그램 내부 데이터 모델로 변환
→ 앱 화면 표시 또는 후속 비즈니스 로직 실행
```

## 핵심 설계 원칙

### 1. AI는 자연어가 아니라 JSON을 반환하게 강제해야 한다

자동화의 핵심은 생성형 AI를 "채팅 도구"가 아니라 "JSON 생성 API"처럼 다루는 것입니다. 프롬프트는 반드시 아래 특징을 가져야 합니다.

- 응답은 앞뒤 설명 없이 JSON 객체 하나만 반환하라고 명시
- 필요한 Key를 미리 고정
- 배열/문자열 규칙을 구체적으로 설명
- 비어 있을 수 있는 값도 명확히 정의

**예시:**

```text
다음 데이터를 분석하세요. 응답은 반드시 마크다운 없이 JSON 객체 하나만 반환하세요.
{
  "analysis_tags": ["", ""],
  "summary": "",
  "is_action_required": false
}
규칙:
- analysis_tags는 2~5개의 짧은 태그입니다.
- summary는 3~5문장 실무 요약입니다.
```

### 2. 웹 화면 전체를 읽으면 안 된다

화면 전체를 복사하면 이전 대화 내용과 UI 텍스트(예: "AI가 타이핑 중...", "프롬프트 추천")가 섞입니다.
응답 추출은 "화면 전체 읽기"가 아니라 **"후보 수집 → 의미 있는 JSON 선별"** 과정으로 가야 안정적입니다.

- 최신 응답 DOM 블록만 특정하여 텍스트를 가져옵니다.
- 후보 텍스트 안에서 정규식 등을 통해 `{ ... }` 형태의 JSON 객체를 모두 찾습니다.
- 찾아낸 객체 중 우리가 요구한 필수 Key(`summary`, `analysis_tags` 등)를 포함한 유의미한(meaningful) 객체만 선별하며, 여러 개일 경우 가장 마지막 객체를 최종 결과로 채택합니다.

### 3. 전송은 한 번으로 끝내지 말고 재시도해야 한다

SPA 환경에서는 붙여넣기 직후 바로 Enter를 누르면 반응하지 않는 경우가 많습니다. 전송 로직은 다음 순서를 복합적으로 사용해야 합니다.

- 입력창 포커스 및 프롬프트 입력
- UI 이벤트 인식을 위한 짧은 대기
- 여러 방식 시도 (Enter, Ctrl+Enter, 전송 버튼 Click 등)
- 각 단계 사이에 지연 시간(Delay) 추가
- **검증 로직:** 입력창에 프롬프트가 아직 남아 있는지 확인하고, 사라졌으면 전송 성공으로 간주합니다.

### 4. 응답이 Placeholder인지 실제 답변인지 구분해야 한다

응답을 너무 빨리 읽으면 로딩 중인 상태 문자열만 잡힐 수 있습니다. 다음 조건으로 완료 상태를 판별합니다.

- 추출된 응답의 길이가 너무 짧지 않은가?
- '생성 중', 'AI' 같은 중간 문구가 아닌가?
- 텍스트 내부에 완전한 JSON 구조가 있는가?
- UI 상단의 '정지' 버튼이 '재생성'이나 '복사' 버튼으로 바뀌었는가?

### 5. 프로그램 내부 객체는 방어적으로 매핑해야 한다

AI가 가끔 지시를 어기고 속성을 누락할 수 있습니다.
최종적으로 프로그램에서 파싱할 때는 없는 속성을 참조하다 뻗지 않도록, 누락된 값은 빈 배열(`[]`)이나 빈 문자열(`""`) 등 기본값(Default)으로 매핑하는 과정이 필수입니다.

## 인증 전략 (세션 유지)

웹 UI 자동화에서 가장 까다로운 부분은 로그인 세션 유지입니다.

### 권장 방식: 전용 자동화 프로필 사용

- 브라우저 제어 툴(Playwright, Selenium)의 `User Data Directory` 기능을 사용해 자동화 전용 프로필 경로를 지정합니다.
- 처음 1회는 사용자가 직접 창을 띄워 수동으로 로그인합니다.
- 이후 스크립트 실행 시 해당 프로필 경로를 재사용하면 로그인 상태(쿠키, 토큰)가 유지됩니다.

### 주의할 점

- 완벽한 백그라운드(Headless) 모드는 봇 탐지 정책에 의해 캡챠가 뜨거나 인증이 잘 유지되지 않는 경우가 많습니다.
- 사내 보안 정책이나 서비스 환경에 따라, 화면이 보이는 **Headed 모드**로 동작시키는 것이 우회하기 가장 쉽고 안정적입니다.

## 실행 단계별 구현 가이드 (8단계)

현재 아키텍처는 아래 8단계로 구현되어야 합니다.

1. **프롬프트 생성:** 시스템 지시문, 메타데이터, 원본 텍스트를 결합하여 하나의 완성된 프롬프트 문자열로 만듭니다.
2. **출력 스키마 결정:** 분석 목적에 맞추어 돌려받을 JSON 스키마를 고정합니다.
3. **AI 페이지 접속:** Playwright 등으로 지정된 프로필을 로드하여 대상 AI 웹서비스(ChatGPT, Claude 등) URL을 엽니다.
4. **입력창 탐지:** 대상 AI의 UI가 일부 바뀌어도 견딜 수 있도록, 아래 selector 후보를 순서대로 확인하여 입력창을 찾습니다.
   - `textarea`
   - `[role='textbox']`
   - `div[contenteditable='true']`
5. **프롬프트 입력 및 전송:** 전송 성공 여부를 아래 흐름으로 판정합니다.
   - 입력 후 잠깐 기다림
   - 여러 방식으로 Enter 및 전송 버튼 클릭 시도
   - 입력창에 프롬프트가 아직 남아 있는지 확인
   - 텍스트가 사라졌으면 전송 성공으로 간주
6. **응답 대기:** 주기적으로 "새 메시지 후보"를 수집하면서 아래 조건을 확인합니다.
   - 이전 화면에 없던 새로운 텍스트인가?
   - 내가 보낸 prompt 자체가 메아리처럼 다시 읽힌 것은 아닌가?
   - 로딩 중임을 나타내는 placeholder(예: "생성 중...")는 아닌가?
   - 텍스트 내부에 JSON 구조가 포함되어 있는가?
7. **응답 추출:** "화면 전체 복사"로 인한 오류를 피하기 위해 아래 로직으로 동작합니다.
   1. 후보 텍스트에서 prompt 부분을 제거
   2. AI가 답변한 본문 텍스트(예: `AI said:` 이후 텍스트)만 우선 격리
   3. 그 안에서 JSON 객체를 여러 개 추출
   4. `analysis_tags`, `summary` 같은 필수 필드를 가진 의미 있는(meaningful) JSON만 채택
   5. 조건에 맞는 객체가 여러 개 발견되면 제일 마지막 객체를 최종 사용
8. **프로그램 데이터로 변환:** 최종 채택된 응답 JSON을 파싱하여, 자동화를 구동하는 메인 언어의 객체(예: Python Dict, C# 클래스 등)로 바꾸고 실제 앱에서 쓰는 분석 모델로 변환합니다.

## 다른 프로그램에 적용하는 방법 (추천 아키텍처)

이 기술은 어떤 애플리케이션이든 아래의 어댑터 패턴으로 적용할 수 있습니다.

```
업무 앱 (메인 로직)
├── Prompt Builder (데이터를 프롬프트로 변환)
├── AI Web Automation Adapter (자동화 오케스트레이션)
│    ├── AI Browser Session (프로필 로드, 브라우저 실행)
│    ├── AI Sender (입력창 탐지, 입력, 전송 검증)
│    └── AI Response Extractor (응답 폴링, JSON 추출)
└── JSON Mapper (추출된 JSON을 업무 객체로 변환)
```

이렇게 브라우저 제어(Adapter) 부분을 모듈로 격리해두면, AI 서비스의 UI가 바뀌어도 해당 모듈의 Selector만 수정하면 됩니다.

## 안정성을 높이기 위한 권장 사항

### 1. 항상 원본 응답(Raw Response)을 로깅한다

자동화는 결국 외부 UI에 의존하기 때문에 파싱 에러 발생 시 원인 추적이 필수입니다. AI가 뱉어낸 날것의 문자열(Raw Response)과 사용된 프롬프트를 반드시 파일이나 DB에 기록해야 합니다.

### 2. JSON 파싱 실패를 정상 시나리오로 간주한다

AI는 지시를 잘 따르다가도 가끔 JSON 앞뒤에 "네, 분석 결과입니다."를 붙이거나 코드 블록을 깨뜨립니다. "파싱 실패는 예외"가 아니라 "자주 일어나는 일상"이므로 정규식 재시도나 억지 파싱 같은 복구 로직을 마련해야 합니다.

### 3. 수동 Fallback 경로는 반드시 남겨둔다

웹 UI 변경, 캡챠, 인증 만료 등으로 자동화 스크립트가 뻗더라도 업무가 멈추면 안 됩니다.
아래의 수동 흐름을 프로그램 내에 반드시 남겨두세요.

1. 프로그램이 프롬프트를 클립보드에 자동 복사.
2. 사용자가 직접 웹 브라우저를 열어 AI에 붙여넣기.
3. 결과를 프로그램에 수동으로 붙여넣기.
4. **파서(Parser)와 매퍼(Mapper) 로직은 동일하게 재사용.**

## 요약 및 현실적인 결론

웹 UI를 제어해 무료로 AI API를 대체하는 프로젝트를 진행할 때 명심해야 할 현실적인 결론입니다.

1. 완전 Headless보다 **Headed 모드**가 봇 탐지 우회에 훨씬 유리하다.
2. 매번 로그인하는 것보다 **전용 자동화 브라우저 프로필**을 재사용하는 것이 압도적으로 안정적이다.
3. 응답을 아예 못 받는 문제보다, 화면의 불필요한 텍스트까지 긁어와서 파싱이 깨지는 문제가 훨씬 흔하다. **(의미 있는 JSON 선별 로직 필수)**
4. 프롬프트의 **JSON 계약을 강하게 잡을수록** 후처리가 쉬워진다.

## 체크리스트 & 추천 구현 순서

이 기술 스택을 새로운 프로젝트에 도입할 때는 아래 순서로 구현하는 것을 강력히 권장합니다.
자동화 봇이 깨지더라도 업무 자체는 진행할 수 있도록 만드는 핵심 순서입니다.

- [ ] 사람이 수동으로 질의할 **프롬프트 템플릿**부터 먼저 안정화한다.
- [ ] 반환받을 **출력 JSON 스키마**를 고정한다.
- [ ] (자동화 없이) 사용자가 복사해온 응답을 처리하는 **파서(Parser)와 비즈니스 로직**부터 구현한다.
- [ ] Playwright 등을 이용한 **자동 입력/전송(Sender)** 로직을 붙인다.
- [ ] 마지막으로 **응답 대기 및 추출(Extractor)** 자동화를 완성한다.
